# TDD strategy

Test-first, always. The discipline that made the Python suite trustworthy carries
over with one rule preserved verbatim: **never mock the browser or CDP — only the
LLM may be faked.**

## Three test tiers

1. **Golden-file unit tests** for pure functions (highest confidence-per-effort).
   Capture the Python implementation's outputs as fixtures and assert the Rust
   output matches byte-for-byte (or field-for-field for JSON):
   - DOM serialization (the pseudo-HTML string) and the selector map.
   - `SchemaOptimizer` output for a corpus of pydantic/JSON schemas.
   - Sensitive-data `<secret>` substitution and TOTP formatting.
   - Loop-detection hashes; element hashing / `MatchLevel`.
   Fixtures live in `crates/<crate>/tests/fixtures/` and are regenerated by
   `xtask capture-fixtures` (which shells into the Python impl).

2. **Live-Chrome integration tests** for stateful crates (`bu-cdp`, `bu-session`,
   `bu-tools`, `bu-mcp`). Boot a real headless Chromium (with `--no-sandbox` on
   CI/Ubuntu 24.04) and exercise real CDP. Gate behind a `live-chrome` cargo
   feature so `cargo test` stays hermetic by default. Mirror the Python fixtures:
   serve local HTML with a tiny in-test HTTP server; never hit remote URLs.

3. **Conformance harness** against the beta JSON-RPC contract
   ([architecture/11-beta-rust-bridge.md](../../architecture/11-beta-rust-bridge.md)).
   `xtask conformance` runs the same task+config through the Python beta bridge and
   the Rust `agent.run_task`, then diffs the emitted `history.events`,
   `usage`, and `success`. This is the end-to-end oracle for the agent loop.

## Fixture capture

`xtask capture-fixtures`:
- Runs `uv run python xtask/capture.py <page-url>` which uses the installed Python
  browser-use to dump: raw CDP `getDocument`/`captureSnapshot`/`getFullAXTree`
  JSON, the fused tree, the serialized DOM string, and the selector map.
- Writes deterministic JSON into the consuming crate's `tests/fixtures/`.
- Committed to the repo so tests are reproducible without a live Python env.

## Per-crate test layout

```
crates/bu-dom/
├── src/
└── tests/
    ├── fixtures/            # captured Python outputs
    ├── serialize_golden.rs # tier-1 golden tests
    └── live_dom.rs         # tier-2, #[cfg(feature = "live-chrome")]
```

## CI

- Default job: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test`
  (tiers 1 only — hermetic).
- Nightly/opt-in job: install Chromium, run `--features live-chrome` (tier 2) and
  `xtask conformance` (tier 3).

## Working agreement for codex

- Red → green → refactor, one crate/seam at a time, in the port order from
  [plan.md](plan.md).
- Do not advance a phase until its exit criterion's tests are green.
- Keep [progress.md](progress.md) updated after each green milestone.
